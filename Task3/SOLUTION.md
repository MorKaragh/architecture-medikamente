# Медикаменте: стратегия защиты данных по состояниям и классам чувствительности

Документ задаёт объём защищаемых данных, модель классификации (состояние, чувствительность, профиль риска), базовый и расширенный набор средств защиты, автоматизированный контроль и реестр программных и аппаратных средств. Операционные потоки и целевой контур согласованы с [`../Task1/SOLUTION.md`](../Task1/SOLUTION.md), [`../Task1/PROBLEMS.md`](../Task1/PROBLEMS.md), [`../Task2/SOLUTION.md`](../Task2/SOLUTION.md).

---

## 1. Какие данные подлежат защите

| Домен | Примеры данных | Почему в зоне защиты |
|-------|----------------|----------------------|
| Идентификация и контакт | ФИО, дата рождения, телефон, email, адрес | ПДн; прямой идентификации субъекта |
| Медицинская тайна и особые категории | диагнозы, назначения, медкарта, результаты анализов, изображения | повышенные требования к конфиденциальности и доступу |
| Финансы и платежи | суммы, чеки, реквизиты операций, токены платёжных инструментов | финансовая тайна и платёжная безопасность |
| Служебные и кадровые | зарплата, кадровые документы, налоговые сведения | ПДн сотрудников, ограниченный круг получателей |
| Технические и поведенческие | логи, IP, идентификаторы сессий, метаданные записей | косвенная идентификация и расследование инцидентов |
| Операционные артефакты | сканы, вложения почты, выгрузки в аналитику | риск накопления копий без lineage и без политики |

## 2. Модель классификации

### 2.1 Состояние данных

| Состояние | Описание | Типовые носители в контуре «Медикаменте» |
|-----------|----------|------------------------------------------|
| **Хранение (at rest)** | Данные в БД, объектном хранилище, бэкапах, журналах на диске | EMR, биллинг, файлы сканов, SIEM-хранилище |
| **Передача (in transit)** | Сетевые и межсервисные каналы | TLS между клиентом и API Gateway, mTLS между сервисами, интеграции с 1С и лабораторией |
| **Обработка (in use)** | Вычисления в памяти приложений, временные файлы, консоли поддержки | бизнес-логика, ETL, сессии администраторов |

Для каждого класса чувствительности политика должна явно покрывать **все три состояния** (шифрование и ключи для at rest и in transit; минимизация времени жизни, изоляция процессов и маскирование для in use).

### 2.2 Уровни чувствительности и соответствие тегам

| Уровень | Смысл | Связка с тегами (`privacy.class`, `pdn.category`) |
|---------|--------|-----------------------------------------------------|
| **P0 — публичный / внутренний неперсональный** | Расписание без ПДн, справочники | `public` / `internal`, `pdn.none` |
| **P1 — персональные (обычные ПДн)** | контакты, запись на приём | `confidential`, `pii` |
| **P2 — финансовые и строго регламентированные** | платежи, чеки, биллинг | `confidential`, `financial` |
| **P3 — медицинские особые категории** | карта, диагнозы, анализы | `restricted-medical`, `special_medical` |

### 2.3 Профиль риска (дополнение к уровню)

| Фактор | Влияние на выбор мер |
|--------|----------------------|
| Юридическое и регуляторное | штрафы, обязанности оператора ПДн, требования к медданным |
| Репутация и доверие пациентов | утечка P3 — максимальный ущерб бренду |
| Масштаб и филиалы | рост поверхности атаки и числа копий данных |
| Зависимость от поставщиков | облако, платёжный провайдер, SaaS SIEM — нужны договорные и технические границы |

## 3. Классификация, обоснование защиты и инструменты

| Данные | В чём необходимость защиты | Инструменты защиты |
|--------|----------------------------|----------------------|
| ПДн пациента (P1) | идентификация, несанкционированный маркетинг/шантаж, нарушение закона о ПДн | TLS 1.2+, шифрование БД/хранилища (AES-256), KMS; RBAC/ABAC; аудит чтения/изменения; маскирование в непривилегированных UI |
| Медицинские особые категории (P3) | медицинская тайна, максимальный вред субъекту | всё для P1 + выделенный контур EMR, MFA, запрет массовых выгрузок без обезличивания; HSM для критичных ключей; DLP на каналах |
| Финансовые и платёжные данные (P2) | мошенничество, PCI-подобные требования к обработке карт | токенизация у платёжного провайдера; не хранить PAN в 1С/Excel; шифрование журналов; сегментация сети от EMR |
| Кадровые и зарплатные (P1–P2) | утечка внутренних сведений, дискриминация | отдельные роли IAM; шифрование; минимизация выгрузок; журналирование |
| Логи и метаданные (P0–P1) | косвенная деанонимизация, компрометация учётных записей | TLS; шифрование at rest; редакция PII/секретов; отдельные учётки для SOC без доступа к EMR |
| Сканы и вложения (P1–P3) | «тихие» копии вне EMR | объектное хранилище с шифрованием и политиками версий; антивирус/CDR на входе; классификация и DLP |
| Потоки в аналитику | обход минимизации и повторная идентификация | ETL с маскированием и псевдонимизацией; витрины только обезличенные; каталог данных и запрет прямых запросов к операционным БД |

## 4. Базовый набор средств (минимум для запуска целевого контура)

| Назначение | Средство (тип) |
|------------|----------------|
| Идентификация и доступ | корпоративный IdP, MFA, RBAC/ABAC |
| Периметр API | API Gateway, WAF, схемы контрактов |
| Криптография | TLS/mTLS, KMS, шифрование томов и БД |
| Секреты | хранилище секретов (vault), ротация |
| Аудит | неизменяемые журналы доступа к P1–P3, интеграция с SIEM |
| Классификация | теги на полях/таблицах/API; матрица доступа |

## 5. Расширяемый набор средств

| Назначение | Средство (тип) |
|------------|----------------|
| Предотвращение утечек | DLP (каналы email, web, съёмные носители, облачные синхронизации — по политике) |
| Поведение пользователей | UEBA поверх SIEM |
| Управление данными | Data Catalog, lineage, DPIA-привязки к витринам |
| Платежи | сервис токенизации / внешний платёжный шлюз |
| Надёжность ключей | HSM для корневых ключей и критичных операций |
| Контейнеры и CI/CD | сканирование образов, policy-as-code, блокировка релиза без тегов данных |

## 6. Автоматизированный контроль за данными

| Механизм | Что контролируется | Как автоматизируется |
|----------|---------------------|----------------------|
| Policy-as-code в CI/CD | новые поля/API без тегов, запрещённые комбинации класса и канала | пайплайн падает при нарушении; отчёт владельцу домена |
| DLP-политики | выгрузки, вложения, копирование в личные облака | правила по шаблонам и меткам; инциденты в SIEM |
| SIEM-корреляции | массовое чтение карты, ночные выгрузки, новые учётки с широкими правами | правила и playbooks; интеграция с IAM-журналами |
| KMS / HSM | жизненный цикл ключей, политики использования | автоматическая ротация, разделение ролей администратора данных и администратора ключей |
| Каталог и запросы | соответствие витрины заявленному классу | автоматические проверки схемы витрины против политики `masking.policy` / `privacy.class` |
| Резервное копирование | целостность и шифрование копий | шифрованные бэкапы, тест восстановления по регламенту |

## 7. Реестр программных и аппаратных средств защиты данных

| № | Наименование / класс продукта | Тип | Назначение в контуре |
|---|------------------------------|-----|----------------------|
| 1 | IdP (OIDC/SAML), MFA | ПО | аутентификация, строгие сессии |
| 2 | API Gateway | ПО | политики трафика, аутентификация вызовов, лимиты |
| 3 | WAF | ПО | защита периметра веб/API |
| 4 | СУБД с TDE / прозрачным шифрованием | ПО | at rest для структурированных данных |
| 5 | KMS | ПО | управление ключами шифрования |
| 6 | HSM | **аппаратное** | защита корневых ключей, операции подписи |
| 7 | Vault / Secret Manager | ПО | секреты приложений и интеграций |
| 8 | Объектное хранилище с SSE | ПО + дисковая подсистема ЦОД | сканы, вложения, бэкапы |
| 9 | SIEM | ПО | корреляция, хранение событий безопасности |
| 10 | DLP | ПО | контроль каналов утечки |
| 11 | Data Catalog / Governance | ПО | классификация, lineage, согласования |
| 12 | Сервис токенизации / платёжный шлюз | ПО (частично внешнее) | минимизация платёжных данных в учётных системах |
| 13 | Антивирус / CDR для файлов | ПО | входящие вложения и сканы |
| 14 | Сетевые экраны, сегментация (NGFW, microsegmentation) | ПО + **сетевое оборудование** | изоляция EMR, биллинга, аналитики |
| 15 | СКЗИ (сертифицированные средства), VPN / ZTNA при удалёнке | ПО + **аппаратное/криптомодули** по требованиям регулятора | каналы администрирования и критичных интеграций |

Конкретные вендоры и редакции выбираются на этапе закупки; таблица задаёт **классы** средств, обязательные для закрытия политик по состояниям данных и уровням P0–P3.

## 8. Сводка

1. Защита строится вокруг **трёх состояний** данных и **четырёх уровней** чувствительности с учётом **профиля риска** (право, репутация, масштаб, поставщики).
2. Базовый контур — **IAM, шифрование (TLS + at rest + KMS), аудит, теги и матрица доступа**; без этого остальные средства не дают предсказуемого эффекта.
3. Расширение — **DLP, UEBA, каталог, токенизация, HSM, жёсткие проверки в CI/CD**; автоматизация снимает человеческий фактор при масштабировании и филиалах.
4. Реестр в разделе 7 фиксирует состав **ПО и аппаратуры** для согласования закупок и границ ответственности.
